Vector databases have become a cornerstone of modern AI applications. Whether you're building a RAG pipeline, a recommendation system, or a semantic search engine, understanding how to deploy and scale vector databases in production is now an essential skill. This post covers the key concepts, popular options, and practical patterns for using vector databases effectively.

## What Are Vector Databases?

Traditional databases store structured data and answer exact queries. Vector databases store high-dimensional numerical representations (embeddings) and answer *similarity* queries: "find me the 10 most semantically similar documents to this input."

Embeddings are generated by models like OpenAI's `text-embedding-3-small`, Cohere's `embed-v3`, or open-source models like `sentence-transformers`. A typical embedding is a float array of 768–3072 dimensions that encodes semantic meaning.

```python
from openai import OpenAI

client = OpenAI()

def embed(text: str) -> list[float]:
    response = client.embeddings.create(
        model="text-embedding-3-small",
        input=text
    )
    return response.data[0].embedding
```

The vector database's job is to index these embeddings so that approximate nearest-neighbor (ANN) queries run in milliseconds, not seconds.

## Choosing the Right Vector Database

There's no single right answer — the best choice depends on your existing stack and scale requirements.

### pgvector (PostgreSQL Extension)

If you're already on Postgres, `pgvector` is the lowest-friction option. It adds a `vector` column type and supports exact and approximate nearest-neighbor search via HNSW and IVFFlat indexes.

```sql
CREATE EXTENSION IF NOT EXISTS vector;

CREATE TABLE documents (
    id SERIAL PRIMARY KEY,
    content TEXT,
    embedding vector(1536)
);

-- Create HNSW index for fast ANN search
CREATE INDEX ON documents USING hnsw (embedding vector_cosine_ops)
WITH (m = 16, ef_construction = 64);

-- Semantic search query
SELECT id, content, 1 - (embedding <=> $1::vector) AS similarity
FROM documents
ORDER BY embedding <=> $1::vector
LIMIT 10;
```

**When to use:** Smaller datasets (<5M vectors), teams that want to keep their data in one place, applications that need to join vector results with relational data.

**Limitations:** Indexing speed degrades at very high scale; no built-in metadata filtering at query time (requires post-filter).

### Qdrant

Qdrant is a purpose-built vector database written in Rust. It excels at filtered vector search — you can filter by payload metadata *before* the ANN search, which is critical for multi-tenant applications.

```python
from qdrant_client import QdrantClient
from qdrant_client.models import Distance, VectorParams, PointStruct, Filter, FieldCondition, MatchValue

client = QdrantClient(url="http://localhost:6333")

# Create collection
client.create_collection(
    collection_name="documents",
    vectors_config=VectorParams(size=1536, distance=Distance.COSINE),
)

# Upsert points with payload
client.upsert(
    collection_name="documents",
    points=[
        PointStruct(
            id=1,
            vector=embed("Hello world"),
            payload={"tenant_id": "acme", "doc_type": "faq"}
        )
    ]
)

# Filtered search — only search within a tenant's data
results = client.search(
    collection_name="documents",
    query_vector=embed("How do I reset my password?"),
    query_filter=Filter(
        must=[FieldCondition(key="tenant_id", match=MatchValue(value="acme"))]
    ),
    limit=5
)
```

### Pinecone

Pinecone is a fully managed, serverless vector database. You get automatic scaling, replication, and metadata filtering without any infrastructure to manage.

```python
from pinecone import Pinecone, ServerlessSpec

pc = Pinecone(api_key="your-api-key")

pc.create_index(
    name="documents",
    dimension=1536,
    metric="cosine",
    spec=ServerlessSpec(cloud="aws", region="us-east-1")
)

index = pc.Index("documents")

# Upsert
index.upsert(vectors=[
    {"id": "doc-1", "values": embed("Hello world"), "metadata": {"source": "faq"}}
])

# Query
results = index.query(
    vector=embed("password reset"),
    top_k=5,
    filter={"source": {"$eq": "faq"}},
    include_metadata=True
)
```

**When to use:** Teams that want zero ops overhead, production workloads that need guaranteed SLAs, or when you're scaling beyond what self-hosted can handle easily.

## Indexing Strategies: HNSW vs IVFFlat

The two dominant ANN index types you'll encounter are:

**HNSW (Hierarchical Navigable Small World):** Builds a multi-layer graph. Offers excellent query latency with high recall, but has a higher memory footprint and slower build time. Best for latency-sensitive applications.

**IVFFlat (Inverted File Index):** Partitions the vector space into clusters. Faster to build and more memory-efficient, but requires choosing `nlist` (number of clusters) carefully. Good for large datasets where you can afford slightly higher latency.

In practice, most production deployments default to HNSW.

## Hybrid Search: Combining Vector and Keyword Search

Pure semantic search can miss exact keyword matches. Hybrid search combines vector similarity with BM25 keyword scoring using Reciprocal Rank Fusion (RRF):

```python
def hybrid_search(query: str, top_k: int = 10) -> list[dict]:
    # Dense (vector) results
    dense_results = vector_search(embed(query), top_k=top_k * 2)
    
    # Sparse (keyword) results  
    sparse_results = bm25_search(query, top_k=top_k * 2)
    
    # Reciprocal Rank Fusion
    scores: dict[str, float] = {}
    k = 60  # RRF constant
    
    for rank, doc in enumerate(dense_results):
        scores[doc["id"]] = scores.get(doc["id"], 0) + 1 / (k + rank + 1)
    
    for rank, doc in enumerate(sparse_results):
        scores[doc["id"]] = scores.get(doc["id"], 0) + 1 / (k + rank + 1)
    
    sorted_ids = sorted(scores, key=scores.get, reverse=True)
    return sorted_ids[:top_k]
```

Qdrant and Weaviate have native hybrid search support; pgvector requires you to implement this yourself using Postgres full-text search alongside vector queries.

## Production Patterns

### Chunking Strategy

How you split documents before embedding has a huge impact on retrieval quality:

- **Fixed-size chunks** (512 tokens with 50-token overlap): Simple, predictable, but can split mid-sentence.
- **Sentence-aware chunks**: Split on sentence boundaries; better semantic coherence.
- **Hierarchical chunks**: Store both paragraph-level and sentence-level embeddings; retrieve paragraphs but rerank on sentences.

### Batch Embedding

Never embed one document at a time in production:

```python
import asyncio
from openai import AsyncOpenAI

async def batch_embed(texts: list[str], batch_size: int = 100) -> list[list[float]]:
    client = AsyncOpenAI()
    all_embeddings = []
    
    for i in range(0, len(texts), batch_size):
        batch = texts[i:i + batch_size]
        response = await client.embeddings.create(
            model="text-embedding-3-small",
            input=batch
        )
        all_embeddings.extend([d.embedding for d in response.data])
    
    return all_embeddings
```

### Monitoring Vector Databases

Key metrics to watch in production:

- **Query latency p99**: ANN queries should stay under 50ms for real-time use cases.
- **Index build time**: Relevant during bulk ingestion or reindexing events.
- **Recall@K**: Periodically evaluate whether your ANN index is returning the right results compared to exact search.
- **Embedding drift**: If you switch embedding models, existing vectors become incompatible — plan for reindexing.

## Conclusion

Vector databases are no longer an exotic technology — they're a standard component in the AI application stack. Start with pgvector if you're on Postgres and have moderate data volumes. Graduate to a purpose-built store like Qdrant or Pinecone when you need advanced filtering, higher throughput, or want to offload infrastructure management.

The most important thing is to nail your chunking strategy and embedding model choice early: both decisions are costly to change once you have data in production. Get those right, implement hybrid search for better recall, and instrument your retrieval quality before you optimize infrastructure.
